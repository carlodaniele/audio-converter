Before <a href="https://wordpress.com/blog/wordpress-7-0/">WordPress 7.0</a>, <a href="https://wordpress.com/blog/wordpress-ai-plugins/">plugins adding AI features</a> had to request and manage an API key to communicate with an AI provider.

This came with several drawbacks in terms of code weight and maintainability. You had to bundle heavy, proprietary SDK libraries into your plugins and write provider-specific cURL requests.

This fragmented scenario has completely changed with the native <a href="https://wordpress.com/blog/wordpress-7-0/#integration-with-ai">AI integration introduced in WordPress 7.0</a>. Today, WordPress features a centralized interface for configuring credentials for various AI providers, along with a new PHP API. This new architecture allows you to send commands, instructions, media files, and data structures seamlessly, while the Core handles the requests.

The new architecture offers substantial benefits for both site admins and plugin developers.

Admins no longer need to configure AI settings across multiple plugins. Starting with WordPress 7.0, you simply install and enable your plugin connectors, enter your API keys into the unified Connectors interface, and they become instantly available to all AI-powered plugins on your site. You can easily monitor AI usage, switch your preferred model, or revoke a provider's access for all plugins with a single click.

For plugin developers, the advantages are even greater, ranging from data security to code portability and a drastic reduction in <a href="https://wordpress.com/blog/scalable-wordpress-agencies/#technical-debt">technical debt</a>.

Ready to start this journey into the new WordPress AI architecture? Let’s dive in.
[toc]
<h2>The three layers of the WordPress AI architecture</h2>
Today, your WordPress sites are natively AI-ready. This is not about out-of-the-box AI features, though, but rather the foundation for building AI-powered websites.

The new architecture is structured across three distinct layers:
[sub-toc count-number="3"]
<h3>AI Connector</h3>
Before <a href="https://wordpress.com/blog/wordpress-7-0/">WordPress 7.0</a>, every plugin that added AI features to your site needed its own system to store and manage your API keys. Your credentials had to be hardcoded into the plugin or managed via a custom settings page. In either case, it was far from an ideal solution for handling sensitive data.

[caption id="attachment_211404" align="alignnone" width="1462"]<img src="https://wordpress.com/wp-content/uploads/2026/06/connectors-screen.png" alt="Connectors screen" width="1462" height="1404" class="size-full wp-image-211404" /> The centralized AI Connectors interface in the WordPress admin dashboard.[/caption]

A unified interface lets you configure all your AI service providers in one central location. This interface brings several major benefits to site management.

From a management and security standpoint, the biggest advantage is that you only need to set up your providers once in the Connectors interface, and you can then forget about them, regardless of how many plugins use them. If you ever need to change your API keys, you won't have to reconfigure every plugin; you can update them all at once in the Connectors screen.

Centralizing credentials also improves your site's security by mitigating the risk of exposing sensitive data through third-party plugins that might be poorly coded or untrustworthy.

Another key advantage is code portability. Your plugins no longer need to know who your AI provider is because the new architecture decouples authentication from plugin logic. This means that if you switch AI providers, your plugin will keep working without requiring a single line of code change.

However, the real breakthrough in portability is the ability for site admins to configure multiple connectors simultaneously. When a plugin triggers a request, WordPress automatically selects the most suitable AI Provider and model to generate the response, based on both the specific <a href="https://make.wordpress.org/core/2026/03/24/introducing-the-ai-client-in-wordpress-7-0/#ai-model-preferences" target="_blank" rel="noopener noreferrer">configuration of the plugin</a> and the capabilities of the available models.

<h3>AI Client</h3>
While the Connectors screen provides the configuration UI for AI Providers, the <a href="https://make.wordpress.org/core/2026/03/24/introducing-the-ai-client-in-wordpress-7-0/" target="_blank" rel="noopener noreferrer">AI Client</a> gives developers the operational tool to communicate with AI models. 

It is a native software interface that standardizes interaction with artificial intelligence models through the global <code>wp_ai_client_prompt()</code> function and a unified set of PHP methods.

<h4>Reduced technical debt</h4>
For developers, a major advantage of the new AI architecture is the reduction of technical debt and dependencies. Before WordPress 7.0, adding AI features came at a very high cost in terms of code maintenance. You had to bundle third-party proprietary SDKs (such as the official libraries for <a href="https://developers.openai.com/api/docs/libraries" target="_blank" rel="noopener noreferrer">OpenAI</a> or <a href="https://platform.claude.com/docs/en/api/client-sdks" target="_blank" rel="noopener noreferrer">Anthropic</a>) and constantly monitor library versions. This resulted in a heavy plugin codebase and opened the door to compatibility errors. 

Furthermore, you had to write many lines of code to handle network logic and response parsing. To make things even worse, the code was vendor-specific, so if you decided to switch AI providers, you would have to rewrite your plugin's entire network architecture.

The AI Client solves this problem at its root by transferring all this heavy work to the WordPress core. Your code becomes completely agnostic to the AI provider; you only need to write your instructions once, and WordPress will translate them into the model's specific "dialect".

<h4>Intelligent model selection</h4>
A powerful feature of the AI Client's Fluent Interface is the ability to set preferred models through the <code>using_model_preference()</code> method, available via the <code>wp_ai_client_prompt()</code> builder function. This method lets you declare an ordered list of models for your plugin's specific task. For example, the following code requests text generation targeting 3 preferred models:

<pre><code class="language-php">$text_result = wp_ai_client_prompt( 'Convert the following transcript into a concise, blog-ready draft.' )
	-&gt;using_model_preference(
		'gemini-2.5-flash',
		'claude-3-5-sonnet',
		'gpt-4o'
	)
	-&gt;generate_text();</code></pre>

WordPress will check the providers configured on the Connectors screen and scan the list of models defined in the plugin's code, selecting the first connector that is also available on the site. If none of the models specified in your code are available among the active Connectors, WordPress will ignore the model preference list and fall back to the first compatible model configured on the site.

This mechanism ensures that your plugin will always work, even if the site admin switches to a different AI provider.

<h3>AI Provider</h3>
An AI Provider is the company that owns, trains, and hosts the Large Language Models (LLMs) we use every day. OpenAI, Google Gemini, and Anthropic are the most common AI providers supported via dedicated connectors in WordPress 7.0, but developers can register additional custom AI Providers.

When your plugin's code calls the <code>generate_text()</code> method — or any other generation method — WordPress compiles and sends a data payload to the AI Provider via the endpoint established by the Connector. Here is the sequence of operations that takes place behind the scenes:
<ul>
	<li>As a first step, the AI Provider receives the input prompt text along with any attached multimedia files (data payload).</li>
	<li>Once received, it processes the request, applying any system instructions.</li>
	<li>From there, it generates the output structured as free text or a JSON object.</li>
	<li>Finally, it sends the output back to WordPress via an HTTP response.</li>
</ul>
Every provider offers a wide variety of models with different capabilities, speeds, and prices. These range from models specialized purely in text processing to multimodal models that accept media files (audio, images, video) alongside a text prompt, to models engineered for specific tasks, such as image generation or speech-to-text conversion.

[caption id="attachment_211411" align="alignnone" width="1718"]<img src="https://wordpress.com/wp-content/uploads/2026/06/gemini-3-models.png" alt="Gemini 3 models" width="1718" height="1534" class="size-full wp-image-211411" /> Google's Gemini 3 model family.[/caption]

Again, the models available to a WordPress site depend on the AI providers the user configures in the Connectors screen. This means your plugin does not dictate which specific provider to use.

<h2>A deep dive into the wp_ai_client_prompt() function</h2>
The <code>wp_ai_client_prompt()</code> function enables plugins to communicate with AI models in a standardized way: you won’t install external libraries, handle provider-specific configurations, or hard-code manual HTTP requests.

It’s a software interface that lets you write your code once, leaving WordPress to handle the heavy lifting and translating your instructions into each model's specific "dialect".

The function returns an instance of the <a href="https://developer.wordpress.org/reference/classes/wp_ai_client_prompt_builder/" target="_blank" rel="noopener noreferrer"><code>WP_AI_Client_Prompt_Builder</code> class</a>, which provides a fluent interface of chainable methods to customize your prompts. The function is <a href="https://core.trac.wordpress.org/browser/tags/7.0/src/wp-includes/ai-client/class-wp-ai-client-prompt-builder.php#L33" target="_blank" rel="noopener noreferrer">described as follows</a>:

<pre><code class="language-php">/**
 * Fluent builder for constructing AI prompts, returning WP_Error on failure.
 *
 * This class provides a fluent interface for building prompts with various
 * content types and model configurations. It wraps the PHP AI Client SDK's
 * PromptBuilder and adds WordPress-specific behavior including WP_Error
 * handling instead of exceptions, snake_case method naming, and integration
 * with the Abilities API.
 *
 * Only the generating methods will return a WP_Error, to not break the fluent
 * interface. As soon as any exception is caught in a chain of method calls,
 * the returned instance will be in an error state, and all subsequent method
 * calls will be no-ops that just return the same error state instance. Only
 * when a generating method is called, the WP_Error will be returned.
 */</code></pre>

<code>wp_ai_client_prompt()</code> does not make an immediate HTTP request; instead, it instantiates and returns a <code>WP_AI_Client_Prompt_Builder</code> object. More than 50 configuration methods populate the object's private properties using a <a href="https://en.wikipedia.org/wiki/Fluent_interface" target="_blank" rel="noopener noreferrer">Fluent Interface</a>, which makes the code highly readable and easy to write.

Here is a non-exhaustive list of methods you will use most frequently:
<ul>
	<li><code>with_text()</code>: Adds text to the current message.</li>
	<li><code>with_file()</code>: Adds a file to the current message.</li>
	<li><code>with_history()</code>: Adds conversation history messages.</li>
	<li><code>using_model_preference()</code>: Sets preferred models to evaluate in order.</li>
	<li><code>using_system_instruction()</code>: Sets the system instruction.</li>
	<li><code>using_max_tokens()</code>: Sets the maximum number of tokens to generate.</li>
	<li><code>using_temperature()</code>: Sets the temperature for generation.</li>
	<li><code>using_top_p()</code>: Sets the top-p value for generation.</li>
	<li><code>using_top_k()</code>: Sets the top-k value for generation.</li>
	<li><code>as_output_file_type()</code>: Sets the output file type.</li>
	<li><code>as_json_response()</code>: Configures the prompt for JSON response output.</li>
	<li><code>is_supported_for_text_generation()</code>: Checks if text generation is supported by the active configuration.</li>
	<li><code>is_supported_for_image_generation()</code>: Checks if image generation is supported by the active configuration.</li>
	<li><code>is_supported_for_text_to_speech_conversion()</code>: Checks if text-to-speech conversion is supported by the active configuration.</li>
	<li><code>is_supported_for_video_generation()</code>: Checks if video generation is supported by the active configuration.</li>
	<li><code>generate_text()</code>: Generates a text result from the prompt.</li>
	<li><code>generate_image()</code>: Generates an image from the prompt.</li>
	<li><code>convert_text_to_speech()</code>: Converts text to speech.</li>
	<li><code>generate_speech()</code>: Generates speech from the prompt.</li>
	<li><code>generate_video()</code>: Generates a video from the prompt.</li>
</ul>

For the complete list of methods, please refer to the <a href="https://github.com/WordPress/wordpress-develop/blob/7.0/src/wp-includes/ai-client/class-wp-ai-client-prompt-builder.php#L33" target="_blank" rel="noopener noreferrer">WordPress core source code</a>.

Suppose you want to accurately transcribe the contents of an audio file. You could use the function as follows:

<pre><code class="language-php">$transcription_builder = wp_ai_client_prompt( $prompt )
	-&gt;with_file( $audio_base64, $mime_type )
	-&gt;using_temperature( 0.1 );

if ( ! $transcription_builder-&gt;is_supported_for_text_generation() ) {
	// No configured provider supports the current prompt configuration.
	return new \WP_Error(
		'no_supported_model',
		'No configured AI model can process this request. Configure a provider in Settings &gt; Connectors.',
		array( 'status' =&gt; 400 )
	);
}

$transcript = $transcription_builder-&gt;generate_text();</code></pre>

Here is what this code does:
<ul>
	<li><code>$prompt</code>: Represents the primary string containing your core instructions.</li>
	<li><code>with_file()</code>: Unlocks the multimodal capabilities of the interface. In the example above, we passed the pre-encoded audio binary payload as a Base64-Encoded string along with its corresponding MIME type.</li>
	<li><code>using_temperature()</code>: Controls the determinism of the response. While a low value (0.1) forces the model to remain faithful to the audio, a higher value allows greater creative freedom. You can combine this method with the <code>using_top_p()</code> (Top-Probability) method for tighter control over the output.</li>
	<li><code>is_supported_for_text_generation()</code>: Checks if the active connectors support the requirements of the current prompt before committing resources to a network call.</li>
	<li><code>generate_text()</code>: This is the terminating method of the Fluent Interface. It sends the unified HTTP request to the active provider and returns either the generated text string or a <code>WP_Error</code> object on failure.</li>
</ul>

In short, we passed the AI Provider a text prompt and an attached audio file. We set a low temperature to ensure the most deterministic result possible, verified that the active configuration supported text generation, and finally executed the speech-to-text transcription based on the AI's response.

With another prompt, we can pass the raw transcript back to the AI to generate a response structured as a JSON object:

<pre><code class="language-php">$structured_json = wp_ai_client_prompt(
	"Convert the following transcript into a concise, blog-ready draft in American English.\n" .
	"Return JSON with a post title and body sections.\n" .
	"Each section must have: heading, level (2 or 3), and paragraphs (array).\n" .
	"If useful, include bullet_points (array) for short actionable lists.\n" .
	"Use short, clear headings and readable paragraph text.\n\n" .
	"Transcript:\n" . $transcript
)
	-&gt;using_system_instruction( "You are a professional WordPress content editor. Always write in American English (use appropriate spelling, grammar, and conventions for that language variant)." )
	-&gt;using_temperature( 0.4 )
	-&gt;as_json_response( $schema )
	-&gt;generate_text();</code></pre>

For this second prompt, we use additional methods to leverage other capabilities of the API:
<ul>
	<li><code>using_system_instruction()</code>: Defines the operational context and behavioral role of the model, cleanly separating its persona ("a professional WordPress content editor") from the data payload.</li>
	<li><code>using_temperature( 0.4 )</code>: Sets a higher temperature than the previous transcription task, allowing the model to use a richer vocabulary and produce a more fluid content.</li>
	<li><code>as_json_response()</code>: Forces the model to return a structured JSON string that matches the layout defined in your <code>$schema</code> variable. This is useful when preparing content for Gutenberg.</li>
</ul>
With an output structured as JSON, you can extract individual fields and programmatically map them into native Gutenberg blocks using core WordPress functions.

For a deeper overview of the AI Client, see also the <a href="https://make.wordpress.org/core/2026/03/24/introducing-the-ai-client-in-wordpress-7-0/" target="_blank" rel="noopener noreferrer">Introducing the AI Client in WordPress 7.0</a> dev note.

<h3>The plugin's data flow</h3>
Before we dive into the code, let’s take a look at how data flows from the WordPress backend (the block editor) through to the PHP server and the AI provider, and then back again.

<a href="#register-api-endpoint">1. Frontend to backend: Asynchronous REST call</a>: the browser sends the audio ID to the plugin’s custom REST endpoint. To prevent the UI from freezing, we register a REST endpoint that allows asynchronous requests to the PHP server.

<a href="#first-ai-pass">2. First AI pass: Speech-to-text transcription</a>: the server intercepts the request, retrieves the audio from the database, and queries the AI client to extract the raw text transcription.

<a href="#second-ai-pass">3. Second AI pass: Content structuring</a>: The server sends the raw transcription back to the AI client and requests that it be formatted as a structured JSON object.

<a href="#block-generation">4. Data hydration and block generation</a>: the JavaScript (JS) script receives the structured JSON object, normalizes it, and maps the fields to native Gutenberg blocks.

Now let's look at the logic behind each phase.

<h3 id="register-api-endpoint">1. Register a custom API endpoint</h3>
The first step in creating the <strong>AI Content Builder</strong> plugin is to register an API endpoint. This is essential because processing an audio file and generating a response can take several seconds (or even minutes) for the AI model. <strong>Registering a REST endpoint enables you to make asynchronous requests via the sidebar controls, preventing the UI from freezing.</strong>

Registering an API endpoint also enables you to make secure AI calls without exposing login credentials by leveraging the WordPress native authentication system. 

The logic for registering and initializing the endpoint is handled in the plugin's PHP file using the two class methods, <code>init()</code> and <code>register_rest_routes()</code>.

<pre><code class="language-php">public static function init(): void {

	// Register the custom REST endpoint used by the sidebar JS.
	add_action( 'rest_api_init', array( __CLASS__, 'register_rest_routes' ) );

	...
}</code></pre>

By hooking the function call into the <code>rest_api_init</code> hook, the plugin only registers the endpoint root when a REST API request is triggered. This improves performance by preventing unnecessary delays in page loading.

You can configure routes using the <code>register_rest_routes()</code> method as shown in the code below:

<pre><code class="language-php">public static function register_rest_routes(): void {
	register_rest_route(
		self::REST_NAMESPACE,
		self::REST_ROUTE,
		array(
			'methods'             =&gt; \WP_REST_Server::CREATABLE, // Alias for POST.
			'callback'            =&gt; array( __CLASS__, 'handle_generate_request' ),
			'permission_callback' =&gt; static function (): bool {
				// Reject unauthenticated or insufficiently privileged requests early.
				return current_user_can( 'edit_posts' );
			},
			'args'                =&gt; array(
				'audio_id' =&gt; array(
					'type'     =&gt; 'integer',
					'required' =&gt; true,
				),
			),
		)
	);
}</code></pre>

This method does the following:

<ul>
	<li>The <a href="https://developer.wordpress.org/reference/functions/register_rest_route/" target="_blank" rel="noopener noreferrer"><code>register_rest_route()</code> function</a> takes three parameters: the route namespace, the route itself, and a configuration array.</li>
	<li><code>WP_REST_Server::CREATABLE</code> is an alias for the POST transport method, which is used to ensure maximum compatibility with the internal architecture of WordPress.</li>
	<li><code>permission_callback</code> is an anonymous function that restricts access to authenticated users who have permission to create and edit their own posts. WordPress then blocks the request before processing it and automatically returns a <a href="https://wordpress.com/blog/401-error/">401 Unauthorized</a> or <a href="https://wordpress.com/blog/403-forbidden-error/">403 Forbidden</a> error in case of unauthorized access.</li>
	<li><code>args</code> is an optional array of additional parameters that can be used to modify the API response. It allows you to specify the data that the request to the endpoint must return.</li>
	<li><code>audio_id</code> is an array of parameters that specifies the type of data that is required. If the returned data type is not an integer or is missing, the REST API stops execution and throws an error.</li>
</ul>
[notice type="info"]Please refer to the official documentation for additional information on <a href="https://developer.wordpress.org/rest-api/extending-the-rest-api/routes-and-endpoints/" target="_blank" rel="noopener noreferrer">routes and endpoints</a>.[/notice]

Finally, ensure the editor sidebar knows where to send the data. To do this, you use the <code>wp_localize_script()</code> function within the <code>enqueue_editor_assets()</code> method.

<pre><code class="language-php">public static function enqueue_editor_assets(): void {
	...

	wp_localize_script(
		'ai-content-builder-editor',
		'AICBData',
		array(
			'restPath' =&gt; '/' . self::REST_NAMESPACE . self::REST_ROUTE,
			'nonce'    =&gt; wp_create_nonce( 'wp_rest' ),
		)
	);
}</code></pre>
<ul>
	<li>The <a href="https://developer.wordpress.org/reference/functions/wp_localize_script/" target="_blank" rel="noopener noreferrer"><code>wp_localize_script()</code> function</a> is primarily used to localize a registered script with data for a JavaScript variable, but here we use it to pass data from PHP to JavaScript via the global <code>AICBData</code> object.</li>
	<li><code>restPath</code> is the endpoint address; specifying it here allows you to avoid hardcoding it in the JavaScript file.</li>
	<li><code>nonce</code> sets a security token to <a href="https://developer.wordpress.org/reference/functions/wp_create_nonce/" target="_blank" rel="noopener noreferrer">prevent unauthorized access</a>.</li>
</ul>
<h3 id="first-ai-pass">2. Accessing the AI Client to extract text from an audio file</h3>
The core of our plugin is the <code>handle_generate_request()</code> method. This method performs a sequence of tasks, beginning with security checks and concluding with the generation of post content to be injected into the block editor.

First, it verifies that the AI Client is available and that AI support has not been disabled on the site.

<pre><code class="language-php">public static function handle_generate_request( \WP_REST_Request $request ) {

<h3 id="second-ai-pass">3. Converting the audio transcript into structured content</h3>
Now that we have the transcript, the plugin must communicate with the AI provider again to obtain a structured JSON response. For this, we need to define a structured array in JSON format.

<pre><code class="language-php">$schema = array(
	'type'       =&gt; 'object',
	'properties' =&gt; array(
		'title'    =&gt; array( 'type' =&gt; 'string' ),
		'sections' =&gt; array(
			'type'  =&gt; 'array',
			'items' =&gt; array(
				'type'       =&gt; 'object',
				'properties' =&gt; array(
					'heading'    =&gt; array( 'type' =&gt; 'string' ),
					'level'      =&gt; array(
						'type' =&gt; 'integer',
					),
					'paragraphs' =&gt; array(
						'type'  =&gt; 'array',
						'items' =&gt; array( 'type' =&gt; 'string' ),
					),
					'bullet_points' =&gt; array(
						'type'  =&gt; 'array',
						'items' =&gt; array( 'type' =&gt; 'string' ),

With this structure, the AI will respond with a JSON object containing a title and an array of sections. Each section must contain a heading, a heading level, at least one paragraph, and bullet points.

Now that the schema is defined, we can move on to configuring the second request to the AI client:

<pre><code class="language-php">$structured_json = wp_ai_client_prompt(
	&quot;Convert the following transcript into a concise, blog-ready draft in $output_language.\n&quot; .
	&quot;Return JSON with a post title and body sections.\n&quot; .
	&quot;Each section must have: heading, level (2 or 3), and paragraphs (array).\n&quot; .
	&quot;If useful, include bullet_points (array) for short actionable lists.\n&quot; .
	&quot;Use short, clear headings and readable paragraph text.\n\n&quot; .
	&quot;Transcript:\n&quot; . $transcript

	-&gt;using_temperature( 0.4 )
	-&gt;as_json_response( $schema )
	-&gt;generate_text();

	return $structured_json;

Let's break down the key points of this code:
<ul>
	<li><code>using_system_instruction()</code> defines the model's persona.</li>
	<li><code>using_temperature( 0.4 )</code> sets a higher temperature value, giving the AI greater creative freedom to generate a response and produce more fluid, rich prose.</li>
	<li><code>as_json_response( $schema )</code> instructs the client to provide the response in a structured format.</li>
	<li>The plugin stops in case of an error.</li>
</ul>
The next step is to decode and sanitize the data.

	<li>The static method <code>normalize_structured_post()</code> sanitizes and normalizes the provider's output before exposing it to JavaScript (see this function's definition on <a href="https://github.com/carlodaniele/ai-content-builder/blob/35ef318c10768969e235c89617a59cc4b6b38eb4/ai-content-builder.php#L476" target="_blank" rel="noopener noreferrer">GitHub</a>).</li>
